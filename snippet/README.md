# Sprint 2 — фильтр полей формы по значению селекта «Тип»

Решение для задачи: на странице
`http://test.amopoint-dev.ru/testzz/testlist.html` присутствует `<select>` с
названием «Тип»; при выборе значения нужно оставить на экране только те поля,
в атрибуте `name` которых встречается выбранная подстрока.

В каталоге три рабочих артефакта и одна демо-страница:

| Файл | Зачем |
|------|-------|
| `testlist.html` | Локальная реплика страницы. Открывается в браузере «двойным кликом», на нижнем `<script>` подцепляет фильтр. |
| `testlist-filter.js` | Подключаемый IIFE-скрипт. Тег `<script src="./testlist-filter.js"></script>` в конце страницы. |
| `testlist-filter.snippet.js` | Готовый к копипасту сниппет для DevTools-консоли. Работает в том числе на чужой удалённой странице. |
| `README.md` | Этот документ. |

## Как использовать

### Вариант 1 — подключить файлом

В конце `<body>` страницы добавьте:

```html
<script src="./testlist-filter.js"></script>
```

Файл — самодостаточный IIFE, без зависимостей. Он:

* находит контролирующий `<select>` (см. ниже алгоритм);
* вешает на него обработчик `change`;
* по изменению значения скрывает / показывает поля;
* поднимает `MutationObserver` на форму, чтобы переприменять фильтр, если
  DOM формы меняется динамически (jQuery-плагины, инлайн-добавление полей и т.д.).

После загрузки доступен дебаг-API:

```js
AmoPointFilter.apply();   // переприменить фильтр (на случай ручной мутации DOM)
AmoPointFilter.reset();   // снять скрытие со всех полей
AmoPointFilter.destroy(); // снять обработчики и MutationObserver
```

Идемпотентность: повторное подключение файла сначала демонтирует предыдущий
экземпляр и только затем поднимает новый. Реализовано через флаг
`window.__amopointTestlistFilter`.

### Вариант 2 — сниппет в консоли DevTools

1. Открыть нужную страницу.
2. Нажать `F12` → вкладка `Console`.
3. Вставить содержимое `testlist-filter.snippet.js`, нажать `Enter`.

Сниппет дополнительно печатает в консоль, какие поля он скрыл и какие
оставил — это удобно при ревью на чужой странице, где быстро понять
структуру `name` атрибутов «на глаз» затруднительно.

## Алгоритм (пошагово)

1. **Находим контроллер.** Ищем первый `<select>` с `name`, удовлетворяющим
   регулярке `/тип|type/i`. Регулярка покрывает и латиницу, и кириллицу,
   потому что в исходной странице поле названо именно «Тип», но при ручной
   проверке на демо удобно работать с `name="type"`.
2. **Определяем «корень» формы.** `controllerSelect.closest('form')` или, если
   формы нет, `document`. Это ограничивает область фильтрации, чтобы не задеть
   соседние формы на странице.
3. **Кэшируем «обёртку» поля.** Для каждого `input/textarea/select[name]` идём
   вверх по DOM, пока не встретим `.form-row`, `.form-group` или `<label>`.
   Если ничего не нашли — обёрткой считаем сам элемент. Результат сохраняется
   в `WeakMap`, чтобы повторные применения фильтра не пересчитывали путь.
4. **На событие `change`** берём `controllerSelect.value`.
   * Если значение пустое → показываем все обёртки.
   * Иначе проходим по `input[name], textarea[name], select[name]` в форме,
     и для каждого проверяем `field.name.includes(value)`.
5. **Скрытие.** Перед первым скрытием запоминаем `wrapper.style.display`
   (обычно пустая строка), затем выставляем `display: none`. При возврате —
   возвращаем сохранённое значение. Это гарантирует, что мы не «съедим»
   inline-стили, которые мог поставить сам разработчик страницы.
6. **MutationObserver.** Слушаем `childList + subtree` на корне формы.
   Если в добавленных узлах находим элементы, попадающие под
   `FIELD_SELECTOR`, — переприменяем фильтр. Это решает кейс,
   когда форма дорисовывается на лету (типичный сценарий в админках на jQuery).
7. **Idempotency / cleanup.** Глобальный флаг `window.__amopointTestlistFilter`
   хранит ссылку на API. При повторном запуске `destroy()` снимает старые
   обработчики и MutationObserver — никакого «двойного срабатывания» нет.

### Сложность

`O(N)` по числу полей с атрибутом `name` под формой при каждом
изменении значения «Типа». Кэш обёрток снижает константу при многократных
переключениях. По памяти — `O(N)` на каждый отображённый элемент.

## Почему vanilla JS, а не библиотека

ТЗ явно разрешает сторонние библиотеки «при условии обоснования». Однако:

* Скрипт подключается к чужой странице — тянуть jQuery / Vue только ради
  переключения видимости полей значило бы добавить 30–80 КБ зависимостей,
  усложнить интеграцию (CDN, версии) и потенциально конфликтовать с уже
  загруженной на странице версией.
* Алгоритм укладывается в ~80 строк vanilla-кода и не нуждается в реактивности
  или шаблонизации.

Поэтому осознанно выбран чистый JS.

## Альтернативные алгоритмы и почему не выбраны

1. **CSS-only через динамический `<style>`** —
   ставим в `<head>` правило вида
   `form *:not([name*="email"]) { display: none }` и подменяем его на `change`.
   * Плюсы: ноль JS-логики, мгновенно.
   * Минусы: нельзя корректно скрыть «обёртку» (`label`, `.form-row`) — нужно
     скрывать именно контейнер, а не сам `input`, иначе остаётся пустой
     `<label>`. Хак с `:has(...)` доступен не во всех браузерах (Firefox ESR
     прошлых версий), и логика «показать всё при пустом значении» через
     `display: none` на родителе ломается. Хрупко при сложных составных
     `name` (`person[email][primary]`).

2. **jQuery `$('[name*="value"]').show()`** —
   классический one-liner.
   * Плюсы: короткий и понятный.
   * Минусы: лишняя зависимость на чужой странице, конфликты версий
     (`$.noConflict()` не всегда применим), и сам по себе не решает задачу
     скрытия именно «обёртки» — пришлось бы дописывать `.closest('.form-row')`,
     то есть выигрыша в коде почти нет. Плюс нет MutationObserver «из коробки».

3. **Переписать форму на Vue / React** — рендерить только нужные поля по
   реактивному стейту.
   * Плюсы: декларативно, расширяемо.
   * Минусы: требует переписать страницу, добавить билд-шаг, заменить
     серверный шаблон. Для одной формы — overkill и нарушение ТЗ
     («сниппет для подключения к существующей странице»).

4. **Server-side: отдельная форма на каждый тип** — рендерить разные шаблоны
   при выборе типа, перезагружая страницу или подгружая по AJAX.
   * Плюсы: вся логика на бэке, чисто, нет JS вообще.
   * Минусы: требует кооперации с бэком, перезагрузку/AJAX-запросы при
     каждом переключении, и в ТЗ задача явно сформулирована как клиентский JS.

5. **Декларативный `data-show-on="email,phone"`** — авторы формы вешают
   на каждое поле атрибут со списком значений «Типа», при которых оно
   показывается, и JS просто читает этот атрибут.
   * Плюсы: O(N), полностью независимо от соглашения по `name`,
     поддерживает поля, чьё имя не содержит подстроку.
   * Минусы: требует менять разметку — а по ТЗ привязка идёт именно к `name`.
     Хороший вариант на будущее, но не подходит «здесь и сейчас».

## Edge cases, которые скрипт обрабатывает

* **Пустое значение `<select>`** (например, опция «— все —» с `value=""`) —
  показываем все поля.
* **Нет `<form>`** — корнем считаем `document`, MutationObserver вешается на
  `document.body` неявно (через `document`).
* **Несколько селектов, попадающих под регулярку** — берём первый (по
  порядку обхода DOM). Если потребуется выбрать другой, можно перед запуском
  сниппета удалить лишний или подменить `TYPE_SELECT_REGEX`.
* **Сам контроллер не должен скрывать сам себя** — мы пропускаем поле, чья
  обёртка совпадает с обёрткой контроллера. Это защищает от случая, когда у
  селекта `name="type"` и при выборе значения `name` он бы попал под подстроку.
* **Inline `display`** на обёртке сохраняется в `WeakMap` и аккуратно
  восстанавливается при `reset()` / `destroy()`.
* **Динамически добавленные поля** — MutationObserver переприменяет фильтр.
* **Повторный запуск** — старый экземпляр демонтируется через `destroy()`,
  дубликатов обработчиков не появляется.
